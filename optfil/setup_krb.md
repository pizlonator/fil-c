# Kerberos Test Setup for Fil-C

This document describes how to set up a test Kerberos realm using the Fil-C-built Kerberos infrastructure.

## Overview

The Fil-C build includes a complete MIT Kerberos implementation:
- **KDC** (Key Distribution Center) - `/opt/fil/sbin/krb5kdc`
- **Admin Server** - `/opt/fil/sbin/kadmind`
- **Client Tools** - `/opt/fil/bin/{kinit,klist,kdestroy,etc.}`
- **Libraries** - `libkrb5.so`, `libgssapi_krb5.so`

All of these are memory-safe, compiled with Fil-C.

## Configuration Files

Kerberos uses standard system locations (not `/opt/fil/etc`):

- **`/etc/krb5.conf`** - Client configuration (used by kinit, SSH, etc.)
- **`/var/lib/krb5kdc/kdc.conf`** - KDC server configuration
- **`/var/lib/krb5kdc/kadm5.acl`** - Admin server ACL
- **`/etc/krb5.keytab`** - Service keys (for sshd, etc.)
- **`/var/lib/krb5kdc/principal*`** - Kerberos database (created by kdb5_util)

## Automated Setup

The `setup_krb.sh` script automates the entire setup:

```bash
sudo ./setup_krb.sh
```

**This script:**
1. Checks that it's running as root
2. Shows what it will do and asks for confirmation
3. **Safety checks** - refuses to overwrite non-test Kerberos configs
4. Creates all necessary config files
5. Initializes the Kerberos database
6. Creates test principals
7. Generates service keytabs

**Safety:** The script only modifies files that were previously created by `setup_krb.sh` (marked with a special comment). It will refuse to touch a production Kerberos installation.

## Manual Setup

If you prefer to set things up manually:

### 1. Create /etc/krb5.conf

```ini
# GENERATED BY FIL-C KRB TEST - Safe to delete
[libdefaults]
    default_realm = TEST.REALM
    default_keytab_name = FILE:/etc/krb5.keytab
    dns_lookup_realm = false
    dns_lookup_kdc = false

[realms]
    TEST.REALM = {
        kdc = localhost
        admin_server = localhost
    }

[domain_realm]
    .localhost = TEST.REALM
    localhost = TEST.REALM
```

### 2. Create /var/lib/krb5kdc/ directory and configs

```bash
sudo mkdir -p /var/lib/krb5kdc
```

**Create /var/lib/krb5kdc/kdc.conf:**

```ini
# GENERATED BY FIL-C KRB TEST - Safe to delete
[kdcdefaults]
    kdc_ports = 88
    kdc_tcp_ports = 88

[realms]
    TEST.REALM = {
        database_name = /var/lib/krb5kdc/principal
        admin_keytab = FILE:/var/lib/krb5kdc/kadm5.keytab
        acl_file = /var/lib/krb5kdc/kadm5.acl
        key_stash_file = /var/lib/krb5kdc/.k5.TEST.REALM
        max_renewable_life = 7d
        max_life = 1d
    }
```

**Create /var/lib/krb5kdc/kadm5.acl:**

```
# GENERATED BY FIL-C KRB TEST - Safe to delete
*/admin@TEST.REALM    *
```

### 3. Initialize the Kerberos database

```bash
sudo /opt/fil/sbin/kdb5_util create -s
```

Enter a master password when prompted (for testing, anything works).

### 4. Create principals

```bash
sudo /opt/fil/sbin/kadmin.local
# In kadmin.local prompt:
addprinc YOUR_USERNAME                    # User principal
addprinc YOUR_USERNAME/admin              # Admin principal (optional)
addprinc -randkey host/$(hostname)        # SSH service principal
ktadd -k /etc/krb5.keytab host/$(hostname)
quit
```

Replace `YOUR_USERNAME` with your actual username.

## Running the KDC and Admin Server

Start the servers in foreground for debugging:

**Terminal 1 - KDC:**
```bash
sudo /opt/fil/sbin/krb5kdc -n
```

**Terminal 2 - Admin Server:**
```bash
sudo /opt/fil/sbin/kadmind -nofork
```

For production, use systemd (see below).

## Testing

### 1. Get a Kerberos ticket

```bash
/opt/fil/bin/kinit YOUR_USERNAME
# Enter the password you set when creating the principal
```

### 2. Verify the ticket

```bash
/opt/fil/bin/klist
```

Should show:
```
Ticket cache: FILE:/tmp/krb5cc_1000
Default principal: YOUR_USERNAME@TEST.REALM

Valid starting       Expires              Service principal
...                  ...                  krbtgt/TEST.REALM@TEST.REALM
```

### 3. Test GSSAPI SSH authentication

Make sure your `/etc/ssh/sshd_config` (or `/opt/fil/etc/ssh/sshd_config`) has:
```
GSSAPIAuthentication yes
GSSAPICleanupCredentials yes
```

Then:
```bash
# Connect using hostname (not "localhost")
ssh -p 10022 $(hostname)
```

Should log in **without asking for a password** (using your Kerberos ticket).

## Systemd Integration (Optional)

To run the Kerberos servers as systemd services:

**Create /etc/systemd/system/filc-krb5-kdc.service:**

```ini
[Unit]
Description=Fil-C Kerberos KDC
After=network.target

[Service]
Type=forking
ExecStart=/opt/fil/sbin/krb5kdc
Restart=on-failure

[Install]
WantedBy=multi-user.target
```

**Create /etc/systemd/system/filc-krb5-admin-server.service:**

```ini
[Unit]
Description=Fil-C Kerberos Admin Server
After=network.target filc-krb5-kdc.service
Requires=filc-krb5-kdc.service

[Service]
Type=forking
ExecStart=/opt/fil/sbin/kadmind
Restart=on-failure

[Install]
WantedBy=multi-user.target
```

Then:
```bash
sudo systemctl daemon-reload
sudo systemctl start filc-krb5-kdc
sudo systemctl start filc-krb5-admin-server
sudo systemctl enable filc-krb5-kdc
sudo systemctl enable filc-krb5-admin-server
```

## Cleanup

To remove the test Kerberos setup:

```bash
sudo rm -rf /etc/krb5.conf /etc/krb5.keytab
sudo rm -rf /var/lib/krb5kdc
```

Only do this if you're sure these are test files!

## Security Notes

**This is a TEST setup:**
- Uses a simple realm name (TEST.REALM)
- Stores the master key in a stash file
- Has permissive admin ACLs
- **Not suitable for production**

For production Kerberos:
- Use your organization's realm name
- Proper master key management
- Restrictive ACLs
- Regular backups of the principal database
- Firewall rules for port 88 (KDC) and 749 (kadmin)

## Troubleshooting

### "Cannot open /etc/krb5.conf"
Make sure the file exists and is readable. Check with:
```bash
ls -la /etc/krb5.conf
```

### "No credentials cache found"
You need to run `kinit` first to get a ticket.

### "Server not found in Kerberos database"
The host principal doesn't exist or is wrong. Check:
```bash
sudo /opt/fil/bin/klist -kt /etc/krb5.keytab
```

Should show `host/YOUR_HOSTNAME@TEST.REALM`.

### SSH asks for password despite having tickets
- Make sure you connect to the **hostname**, not "localhost"
- Check that sshd has `GSSAPIAuthentication yes`
- Verify the keytab matches the hostname:
  ```bash
  ssh -p 10022 $(hostname)  # Use hostname, not localhost
  ```

### "Clock skew too great"
Kerberos is sensitive to clock differences. Make sure system time is correct:
```bash
sudo ntpdate pool.ntp.org  # Or use systemd-timesyncd
```
